# CC Poker 개발 로드맵 (v1.0)

## 1) 프로젝트 목표
- 목표: 브라우저에서 동작하는 무료 온라인 텍사스 홀덤(우선 NLH) 플랫폼 구축
- 핵심 방향:
  - `서버 권위(Server-authoritative)` 구조로 공정성 확보
  - `고품질 UX` (반응성, 애니메이션, 직관적 테이블 경험)
  - `확장 가능한 아키텍처` (동시 접속 증가 대응)
  - `운영 가능한 제품` (모니터링, 장애 대응, 배포 자동화)

## 2) 제품 범위 (MVP -> 확장)

### MVP (반드시 포함)
- 회원/게스트 진입, 닉네임 기반 플레이
- 2~9인 캐시 테이블(무료 칩, 최소 2명 시작)
- 핵심 게임 플로우:
  - 블라인드/딜/베팅 라운드(Preflop/Flop/Turn/River)
  - 폴드/체크/콜/베트/레이즈/올인
  - 쇼다운 및 팟 분배(사이드팟 포함)
- 실시간 동기화 (WebSocket)
- 기본 재연결/타임아웃 처리
- 최소 운영 기능:
  - 테이블 생성/종료
  - 간단 관리자 로그/상태 확인

### Post-MVP (2차)
- 친구 초대/프라이빗 테이블
- 관전자 모드
- 핸드 히스토리/리플레이
- 랭크/시즌(실제 상금 없는 경쟁 요소)
- 토너먼트 모드(SNG -> MTT 순)

### 장기 (3차+)
- 모바일 PWA 최적화
- 다국어(i18n)
- 고급 안티치트(행동 패턴 이상 탐지)
- 리더보드/업적/클랜 시스템

## 3) 기술 아키텍처

### 백엔드 (Go)
- 언어/런타임: Go 1.23+ (프로젝트 시작 시 최신 안정판 기준)
- API:
  - REST: 인증, 로비, 프로필, 관리 기능
  - WebSocket: 테이블 입장/행동/브로드캐스트
- 권장 구조:
  - `cmd/server` (엔트리포인트)
  - `internal/app` (DI, 서비스 초기화)
  - `internal/game` (포커 룰 엔진/상태머신)
  - `internal/table` (테이블 라이프사이클)
  - `internal/ws` (실시간 세션/허브)
  - `internal/store` (DB/캐시 접근)
  - `pkg/protocol` (공유 이벤트 스키마)
- 상태 관리:
  - 메모리 기반 테이블 런타임 + 주기적 스냅샷/이벤트 로그
  - 테이블 단위 싱글 스레드(Actor-like) 처리로 레이스 최소화

### 프론트엔드 (React + Phaser)
- React: 로비/테이블 UI, 상태 관리, 접근성
- Phaser: 카드 이동, 칩 애니메이션, 딜/팟 연출
- 원칙:
  - 게임 룰 계산은 서버만 수행
  - 클라이언트는 표현과 입력 전달에 집중
- 권장 구성:
  - `web/src/features/lobby`
  - `web/src/features/table-ui` (React)
  - `web/src/features/table-scene` (Phaser)
  - `web/src/shared/net` (WS/REST 클라이언트)
  - `web/src/shared/state` (Zustand/Redux 중 택1)

### 데이터/인프라
- DB: PostgreSQL (계정, 지갑, 히스토리)
- 캐시/큐: Redis (세션, 임시 상태, 레이트리밋)
- 배포:
  - 컨테이너 기반 (Docker)
  - CI/CD (lint/test/build/deploy)
  - 모니터링: Prometheus + Grafana + 중앙 로그 수집

## 4) 핵심 설계 원칙
- 서버 권위형 판정: 클라이언트 조작으로 승패가 바뀌지 않게 설계
- 결정론적 게임 엔진: 같은 입력 -> 같은 결과 보장
- 이벤트 프로토콜 버전 관리: `event_version`으로 호환성 유지
- 장애 허용성: 플레이어 이탈/재접속, 서버 재기동 대비
- 관측 가능성: 핸드 단위 트레이싱, 테이블 지표, 오류 추적

## 5) 단계별 로드맵

## Phase 0. 기획 확정 (1~2주)
### 목표
- 룰/정책/우선순위 확정 및 문서화

### 작업
- 게임 정책 정의:
  - 테이블 인원 정책(최소 2명, 최대 9명)
  - 스택, 블라인드 구조, 타임뱅크(기본 30초 + 잔여 누적 60초), 자리 비우기 정책
- 인증 방식 확정:
  - WebSocket/REST 모두 HttpOnly 세션 쿠키 기반으로 통일
- 비기능 요구사항 정의:
  - 목표 지연시간(p95), 동시 테이블 수, 가용성
- 데이터 모델 초안
- 이벤트 스키마 초안 작성(클라/서버 공통)
- 개발 컨벤션 확정(Go 코드 한국어 주석 규칙 포함)

### 완료 기준 (DoD)
- PRD + 아키텍처 초안 + 이벤트 명세 문서 승인
- Go 개발 컨벤션 문서 승인

## Phase 1. 개발 기반 구축 (1~2주)
### 목표
- 팀이 바로 기능 개발할 수 있는 뼈대 완성

### 작업
- 모노레포 구조/패키지 매니저/빌드 규칙 확정
- 코드 품질 체계:
  - Go lint/test, ESLint/TypeScript, Prettier
- 기본 CI 파이프라인 구축 (PR마다 실행)
- 로컬 개발환경:
  - Docker Compose (Postgres/Redis)
- 공통 에러 포맷/로깅 규약 설정

### 완료 기준 (DoD)
- `make dev` 또는 동등 커맨드 1회로 로컬 기동
- PR 시 자동 테스트/린트 성공

## Phase 2. 포커 엔진 구현 (2~4주)
### 목표
- 서버에서 완전한 핸드 진행 가능

### 작업
- 카드/덱/핸드 랭킹 엔진
- 베팅 라운드 상태머신
- 올인/사이드팟 정산
- 시트/버튼/블라인드 로테이션
- 타이머/액션 제한시간 처리

### 완료 기준 (DoD)
- 엔진 단위 테스트 커버리지 고도화
- 재현 가능한 시나리오 테스트 (고정 시드)
- 잘못된 액션 입력 방어

## Phase 3. 실시간 테이블 서버 (2~3주)
### 목표
- 다수 플레이어가 같은 테이블에서 실시간 플레이

### 작업
- WebSocket 연결/인증/세션관리
- 테이블 매니저 + 룸 브로드캐스트
- 재연결 복구:
  - 핸드 중 재접속 시 상태 복원
- 관전자 이벤트 채널(초기 버전)
- 메시지 ACK/시퀀스 전략 정의

### 완료 기준 (DoD)
- 2~9인 시뮬레이션에서 동기화 오류 없음
- 재접속 후 상태 불일치율 0에 수렴

## Phase 4. 프론트엔드 테이블 UX + Phaser 연출 (3~4주)
### 목표
- 플레이 가능한 고품질 게임 테이블 UI 완성

### 작업
- React 테이블 레이아웃(좌석/액션패널/팟/히스토리)
- Phaser 씬 통합:
  - 카드 딜/플립
  - 칩 이동/팟 모으기
  - 승리 하이라이트
- 네트워크 지연 상태 UI (로딩/재접속/복구)
- 사운드/피드백(선택적 토글)

### 완료 기준 (DoD)
- UX 기준 충족:
  - 입력-반응 지연 체감 최소화
  - 모바일/데스크톱 레이아웃 깨짐 없음

## Phase 5. 계정/로비/매치 흐름 (2~3주)
### 목표
- 유저가 자연스럽게 입장하고 계속 플레이 가능한 구조 완성

### 작업
- 계정/게스트 모드
- 로비 목록/테이블 입장/퇴장
- 칩 지급 정책(무료칩 리필)
- 간단 프로필(닉네임, 아바타 기본값)
- 기본 제재 정책(욕설/악성 유저 대응 최소 기능)

### 완료 기준 (DoD)
- 신규 유저 온보딩 -> 플레이 -> 재입장 흐름 완결

## Phase 6. 품질 고도화 (2~3주)
### 목표
- 출시 가능한 안정성/보안/성능 확보

### 작업
- 테스트 강화:
  - Go 단위/통합
  - 프론트 E2E (핵심 유저 플로우)
- 부하 테스트:
  - 동시 접속/테이블 수 목표치 검증
- 보안:
  - 레이트리밋, 입력 검증, 세션 쿠키 만료 정책
- 운영 대시보드:
  - 테이블 수, 접속자, 오류율, 응답시간

### 완료 기준 (DoD)
- SLO 기준 통과
- 주요 치명 버그 0건

## Phase 7. 베타 오픈 및 개선 루프 (지속)
### 목표
- 실제 사용자 피드백 기반으로 빠르게 개선

### 작업
- 클로즈드 베타(소규모)
- 이벤트 로그 기반 UX 병목 분석
- 주간 릴리즈 사이클 운영
- 우선순위:
  - 끊김/동기화/불공정 이슈 최우선

### 완료 기준 (DoD)
- 재방문율/세션 길이 등 제품 지표 개선 추세 확인

## 6) 테스트 전략
- 백엔드:
  - 엔진 규칙 테스트(핵심)
  - 프로토콜 직렬화/역직렬화 테스트
  - WS 통합 테스트(가짜 클라이언트 N개)
- 프론트:
  - 상태 관리 테스트
  - 핵심 컴포넌트 상호작용 테스트
  - E2E: 입장 -> 플레이 -> 쇼다운 -> 재접속
- 부하/혼돈 테스트:
  - 네트워크 지연, 패킷 순서 뒤바뀜, 강제 연결 끊김

## 7) 보안/공정성 체크리스트
- 서버에서만 카드/덱 생성 및 셔플
- 클라이언트 입력은 허용된 액션만 검증 후 반영
- 난수 시드 관리 및 로그 추적 가능성 확보
- 이상 행동 감지:
  - 비정상 속도 액션
  - 비정상 반복 패턴
- 감사 로그:
  - 핸드별 액션 타임라인 저장

## 8) 협업/운영 규칙
- 브랜치 전략:
  - `main` 안정
  - `develop` 통합
  - `feature/*` 작업
- PR 규칙:
  - 작은 단위, 테스트 포함
  - 리뷰 체크리스트 필수
- 문서 동기화:
  - 기능 추가 시 `docs` 내 명세 갱신

## 9) 제안 일정 (예시: 14~20주)
- 1~2주: Phase 0
- 3~4주: Phase 1
- 5~8주: Phase 2
- 9~11주: Phase 3
- 12~15주: Phase 4
- 16~18주: Phase 5
- 19~20주: Phase 6
- 이후: Phase 7 반복

## 10) 즉시 실행할 다음 액션 (이번 주)
1. 게임 정책 문서 확정 (인원 2~9명/블라인드/타이머 30초+타임뱅크 60초 누적/재접속 정책)
2. 이벤트 스키마 초안 작성 (`player_action`, `state_snapshot`, `hand_result`)
3. 모노레포 초기화 (Go 서버 + React 앱 + 공통 프로토콜 패키지)
4. 로컬 인프라 도커 구성 (Postgres/Redis)
5. 포커 엔진 최소 루프(덱/딜/액션/쇼다운) 스파이크 구현
6. Go 한국어 주석 컨벤션 확정 및 PR 체크리스트 반영
