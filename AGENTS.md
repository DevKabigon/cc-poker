# CC Poker Backend Agent Rules (Go)

## 목적
- 백엔드 구현 시 품질/일관성을 유지하기 위한 프로젝트 규칙이다.
- 이 문서의 규칙은 Go 서버 코드 작성/리뷰/리팩터링에 공통 적용한다.

## 해야 하는 규칙 (Do)
1. 게임 판정은 반드시 서버 권위형으로 구현한다.
2. 테이블 상태는 테이블 단위 직렬 처리 루프(Actor-like)로 다룬다.
3. 모든 클라이언트 입력은 `인증 -> 스키마 검증 -> 도메인 검증` 순으로 검증한다.
4. 공개 식별자(타입/함수/메서드)와 복잡한 도메인 로직에는 한국어 주석을 작성한다.
5. 에러는 맥락을 포함해 래핑한다. (`fmt.Errorf(\"...: %w\", err)`)
6. 로그에는 최소한 `table_id`, `hand_id`, `player_id` 중 가능한 식별자를 포함한다.
7. 승패/칩/사이드팟/타임아웃 로직에는 단위 테스트를 반드시 작성한다.
8. WebSocket 인증은 HttpOnly 세션 쿠키 기반으로 구현하고 Origin 검증을 적용한다.
9. 타이머 정책은 기본 30초 + 타임뱅크 60초 잔여 유지 규칙을 따른다.
10. 시간/랜덤/외부 의존성은 인터페이스로 분리해 테스트 가능하게 만든다.

## 해서는 안 되는 규칙 (Don't)
1. 클라이언트에서 승패/칩 계산 결과를 신뢰하거나 그대로 반영하지 않는다.
2. 런타임 제어 흐름에 `panic`을 사용하지 않는다. (초기화 실패 제외)
3. 에러를 무시하거나 로그만 남기고 정상 처리한 것처럼 넘기지 않는다.
4. 테이블 간 공유 mutable 상태를 잠금 없이 직접 수정하지 않는다.
5. 이벤트 시퀀스(`seq`) 역전/누락을 무시하지 않는다.
6. 하드코딩된 매직 넘버를 도메인 로직에 직접 박아 넣지 않는다.
7. 요청 컨텍스트(`context.Context`)를 무시한 DB/네트워크 호출을 작성하지 않는다.
8. 비밀값(토큰/키/비밀번호)을 코드/로그/테스트 데이터에 커밋하지 않는다.
9. 사용자 식별 정보가 포함된 민감 로그를 마스킹 없이 남기지 않는다.
10. 한 함수에 다수 책임(검증+정산+브로드캐스트+저장)을 몰아 넣지 않는다.

## PR 체크 항목 (백엔드)
1. 한국어 주석 규칙을 만족하는가?
2. 서버 권위형 원칙을 위반한 코드가 없는가?
3. 예외/에러 경로 테스트가 포함됐는가?
4. 동시성 위험(레이스/교착) 가능성을 점검했는가?
5. 로그/메트릭으로 장애 분석이 가능하게 작성했는가?

## 문서 운영 규칙
1. `docs/work-logs`에는 에이전트가 수행한 작업을 간단히 요약한 md 파일을 남긴다.
2. work-log 파일명 규칙은 `work + 날짜 + 시간(일본시각) + 번호`를 따른다.
3. 권장 포맷 예시는 `work-YYYYMMDD-HHMM-01.md`로 한다.
4. `docs/commits`는 사용자가 "커밋해줘"라고 요청한 경우에만 작성한다.
5. commit 파일명 규칙은 `commit + 날짜 + 시간(일본시각) + 커밋메세지`를 따른다.
6. 권장 포맷 예시는 `commit-YYYYMMDD-HHMM-커밋메세지.md`로 한다.
